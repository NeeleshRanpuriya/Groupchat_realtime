Status: Pending

1. INTRODUCTION
Problem formalization is the most critical foundation stage of this project because it transforms a broad social issue into an implementable engineering problem. In this project, the social issue is unhealthy online communication, including toxic language, harassment, sarcasm-driven conflict, and emotionally aggressive behavior in real-time chat rooms. Without formalization, the project would remain an idea; with formalization, it becomes a measurable system.

2. CONTEXT OF THE PROBLEM
Digital communities on social media, educational groups, gaming servers, and collaborative team platforms suffer from communication quality problems. Messages are posted quickly, often without self-filtering, and harmful language spreads before manual moderation can react. Existing moderation methods are usually delayed, binary (delete/keep), and not supportive. They do not guide users toward better language. Therefore, there is a need for a system that is:
- Real-time
- Assistive (not only punitive)
- Data-driven
- Scalable across rooms and users

3. FORMAL PROBLEM STATEMENT
Given a message stream M = {m1, m2, m3, ...} generated by users U = {u1, u2, ...} in chat rooms R = {r1, r2, ...}, design and implement a system F that maps each incoming message m at time t into a structured moderation output O.

Mathematically:
F(m, u, r, t) -> O

Where O includes:
- Toxicity score T in [0,1]
- Toxic flag B in {0,1}
- Intent class I from finite set C_intent
- Tone class N from finite set C_tone
- Coaching message C_text
- Suggested rewrite R_text (if required)

4. PRIMARY OBJECTIVES
The main objectives are formalized as follows:
- Objective O1: Detect toxic content with confidence scoring.
- Objective O2: Classify communicative intent (question, complaint, insult, threat, etc.).
- Objective O3: Identify tone quality (polite, neutral, rude, aggressive, passive-aggressive, sarcastic).
- Objective O4: Generate constructive coaching for communication improvement.
- Objective O5: Suggest polite rewrites for harmful statements.
- Objective O6: Return the full moderation result in real-time during chat.

5. FUNCTIONAL REQUIREMENTS
FR1. User messages must be accepted from a real-time channel.
FR2. Every message must pass through moderation pipeline before final broadcast metadata is produced.
FR3. System must persist message plus moderation attributes in database.
FR4. Toxic messages should trigger coaching and rewrite generation.
FR5. APIs should provide message history and moderation analytics.
FR6. Room and user-level controls must be supported for practical deployment.

6. NON-FUNCTIONAL REQUIREMENTS
NFR1. Latency: moderation response should be low enough to maintain live chat usability.
NFR2. Reliability: service must not crash when optional AI APIs are unavailable; fallback logic is required.
NFR3. Usability: moderation output should be understandable to end users.
NFR4. Maintainability: modular architecture for detector, classifier, analyzer, and APIs.
NFR5. Extensibility: future support for multilingual models and stricter policy engines.

7. SCOPE BOUNDARIES
In-scope:
- Real-time moderation for text messages
- Toxicity detection and score estimation
- Intent and tone classification
- Coaching and rewrite support
- Storage, retrieval, and room-based communication

Out-of-scope for current phase:
- Full enterprise distributed microservices environment
- Legal compliance automation for all jurisdictions
- Audio/video moderation at production grade
- Human moderation queue management dashboard with SLA controls

8. KEY ASSUMPTIONS
- Users communicate primarily via short text messages.
- Toxicity can be estimated from linguistic patterns and transformer outputs.
- Intent classes selected are sufficient for baseline moderation quality.
- API-based tone generation is available or fallback heuristics can replace it temporarily.

9. CONSTRAINT ANALYSIS
- Compute constraints: transformer model inference may be heavy on low-resource systems.
- Data constraints: custom datasets are limited in size; generalization may vary.
- API constraints: external service limits and key availability.
- Language constraints: code-mixed Hinglish and obfuscated abusive spellings require normalization and patterns.

10. RISK IDENTIFICATION
R1. False positives: neutral messages flagged as toxic.
R2. False negatives: harmful messages not detected.
R3. Context loss: isolated message classification may miss conversation history.
R4. API downtime: tone/rewrite service interruptions.
R5. User resistance: users may ignore coaching if tone feels robotic.

11. SUCCESS CRITERIA
SC1. Every incoming message receives moderation metadata.
SC2. Toxic content detection accuracy is practically useful in demo scenarios.
SC3. Coaching appears for harmful messages and remains constructive.
SC4. Room chat remains functional under real-time traffic.
SC5. Stored data supports later analytics and interpretation.

12. DELIVERABLES OF THIS MILESTONE

13. CONCLUSION
Problem formalization converts the project from concept to executable design intent. This stage establishes what the system must solve, what boundaries it should respect, and how success should be measured. A strong formalization ensures that complete design, coding, report writing, and final project integration remain aligned to the same technical objective and academic outcome.
