1. INTRODUCTION
Complete design translates the formalized problem into architecture, modules, data contracts, control flow, and implementation-ready specifications. This stage is where abstract requirements become concrete structures that developers can code and test.

2. DESIGN GOALS
DG1. Build a modular, understandable, and maintainable system.
DG2. Support real-time chat without blocking user experience.
DG3. Integrate AI moderation components as independent services.
DG4. Preserve moderation transparency through persistence and APIs.
DG5. Enable iterative enhancement without full rewrite.

3. ARCHITECTURE OVERVIEW
The project uses a layered full-stack architecture:
- Presentation Layer: Next.js frontend for user interaction.
- Communication Layer: REST APIs + WebSocket channels.
- Application Layer: FastAPI business logic and orchestration.
- Intelligence Layer: Toxicity, intent, tone, coaching, and rewrite modules.
- Persistence Layer: SQLAlchemy-backed relational database.

4. HIGH-LEVEL COMPONENTS
4.1 Frontend Components
- Authentication pages (login/register)
- Home and room navigation pages
- Chat room page for real-time messaging
- Analysis panels and moderation indicators
- Group-level controls (members, requests, pinned messages)

4.2 Backend Components
- Auth service: token generation and user access control
- Chat service: message handling and room-level events
- Moderation pipeline coordinator
- Room administration APIs
- Upload/media endpoint integration
- Health and analytics endpoints

4.3 AI Components
- Toxicity detector using transformer model and text normalization
- Intent classifier using pattern rules and category mapping
- Tone analyzer with OpenAI API plus fallback strategy
- Coaching and rewrite generator for harmful tone correction

5. DATA FLOW DESIGN
Step 1: User submits message from UI.
Step 2: Backend receives message through WebSocket.
Step 3: Message is normalized and passed to moderation pipeline.
Step 4: Toxicity, intent, and tone outputs are generated.
Step 5: Coaching and rewrite are conditionally generated.
Step 6: Structured message object is saved to database.
Step 7: Broadcast payload is sent to room participants.
Step 8: Analytics endpoints can later consume persisted data.

6. DATABASE DESIGN
6.1 Core Entities
User:
- user identity, credentials, profile metadata

ChatMessage:
- original message plus moderation attributes
- room id and timestamp for time-series analysis
- edit/pin/reply/file metadata for rich interaction

MessageReaction:
- emoji reactions linked to message id and username

6.2 Design Rationale
- Normalized entity separation keeps logic clean.
- ChatMessage acts as canonical moderation record.
- Reaction model remains lightweight and query-friendly.

7. API DESIGN APPROACH
- REST endpoints for deterministic operations (auth/profile/history/stats).
- WebSocket for real-time bidirectional room messaging.
- JSON payload contracts to simplify frontend integration.
- Route-level access checks for secure room administration.

8. SECURITY DESIGN
- Password hashing with bcrypt flow.
- JWT access tokens for authenticated APIs.
- Protected endpoints for user/account/room actions.
- File-type validation in upload workflow.
- Scope for future enhancements: role-based authorization and rate limiting.

9. PERFORMANCE AND SCALABILITY DESIGN
- Asynchronous request handling in FastAPI.
- Real-time delivery via persistent WebSocket channels.
- Model invocation optimized with fallback logic when needed.
- Scalable path: move to managed database + distributed websocket strategy.

10. ERROR HANDLING DESIGN
- Input validation through schemas.
- Defensive fallback for unavailable AI services.
- Structured HTTP exceptions for API clarity.
- Client-visible status updates for failed operations.

11. UI/UX DESIGN DECISIONS
- Real-time feedback panels to build moderation transparency.
- Dark mode support and responsive layout.
- Action-oriented components for moderation workflows.
- Message-level interactions (edit, pin, react, reply) improve usability.

12. DESIGN VALIDATION CRITERIA
- Can every requirement map to one or more modules?
- Are data entities sufficient for feature set and analytics?
- Does the message lifecycle support real-time constraints?
- Can services fail gracefully without total system failure?
